# 1. Overview

NourishOS uses **Cloud Firestore** as its primary database.

The database is designed to be:

- Document-oriented
- Scalable
- Secure
- Multi-outlet ready
- Offline-capable
- Optimized for real-time collaboration

Data is organized into **collections** and **documents**, with selective use of subcollections where it improves performance and simplifies access patterns.

---

# 2. Design Principles

- Prefer flat collections over deeply nested structures.
- Duplicate small amounts of data when it improves read performance.
- Avoid joins; design documents around application queries.
- Keep document size well below Firestore's 1 MiB limit.
- Use Cloud Functions for business logic and data consistency.
- Enforce security with Firestore Security Rules and RBAC.

- plural nouns

- Auto-generated by Firestore unless a business identifier is required.

# 20. File Storage

Binary files are stored in **Firebase Cloud Storage**.

Firestore stores only metadata:

```json
{
  "fileName": "contract.pdf",
  "fileUrl": "",
  "contentType": "application/pdf",
  "size": 102400,
  "uploadedBy": "",
  "uploadedAt": "Timestamp"
}
```

# 21. Multi-Outlet Strategy

Every operational document should include:

- outletId
- departmentId

Cloud Functions and Firestore Security Rules must restrict data access based on the user's assigned outlet and role, unless elevated permissions allow cross-outlet visibility.

---

# 22. Firestore Indexing

Create composite indexes for common queries, such as:

- tasks by assignedTo + status
- dailyReports by outletId + date
- notifications by userId + isRead
- expenseRequests by status + createdAt
- workOrders by assignedTo + priority

Additional indexes should be added as new query patterns emerge.

---

# 23. Security Model

Access control is enforced through:

- Firebase Authentication
- Firestore Security Rules
- Role-Based Access Control (RBAC)
- Cloud Functions for sensitive operations

Clients should never directly modify approval states, audit logs, or role assignments.

---

# 24. Data Lifecycle

- Active records remain in primary collections.
- Soft deletion is preferred using `isArchived`.
- Scheduled Cloud Functions may archive or purge data based on retention policies.
- Audit logs remain permanent unless legal requirements dictate otherwise.

---

# 25. Backup and Recovery

- Enable automated Firestore exports to Cloud Storage.
- Schedule regular backups (daily recommended).
- Periodically test restore procedures in a staging environment.
- Version critical documents such as SOPs and templates.

# 27. Database Principles

- Design around application queries.
- Keep documents focused and cohesive.
- Minimize deep nesting.
- Use Cloud Functions for business logic.
- Secure every collection with least-privilege access.
- Favor consistency, maintainability, and scalability over premature optimization.
